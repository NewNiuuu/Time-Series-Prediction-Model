## 二、模块分层设计思想

### 第一层：数据处理层 — `data_processor.py`

**解决的问题**：原始业务数据（Excel账单）是"脏"的、杂乱的，不能直接喂给模型。

**设计思想**：

| 真实痛点 | 解决思路 |
|---------|---------|
| 不同文件表头位置不同（有的在第15行，有的在第17行） | 逐行搜索"交易时间"关键词，动态定位真实表头 |
| 不同年份列名不一致（"金额" vs "金额(元)"） | 两阶段列名映射：精确匹配 → 模糊兜底 |
| 原始数据是逐笔交易，稀疏不规整 | 按天重采样（sum聚合），缺失日期补0 |
| 营业额量纲差异大（几百~几千元） | MinMaxScaler [0,1] 归一化，消除量纲 |

**核心设计哲学**：**数据与模型解耦**。数据处理器只负责"把数据弄干净"，不关心后续是 LSTM 还是 Transformer。

---

### 第二层：数据供给层 — `dataset.py`

**解决的问题**：如何把规整的时间序列数据，转化为 PyTorch 训练所需的批量 Tensor。

**设计思想**：

| 核心概念 | 解决思路 |
|---------|---------|
| 时序数据的"监督学习"格式 | 滑动窗口：用前14天预测第15天，窗口每次滑动1格 |
| PyTorch 的数据抽象 | 实现 `Dataset` 协议（`__len__` + `__getitem__`），PyTorch 自动处理 batching/shuffle |
| 时间序列的严格顺序 | `shuffle=False`，按时间顺序 70/15/15 切分，绝不打乱 |

**关键数值**：窗口长度 14 → 183天数据 → 169 个样本。损失了14个样本但换来了规整的监督学习格式。

**核心设计哲学**：**数据管道的可替换性**。未来换股票数据时，只需修改 `data_processor.py` 的列名映射，`dataset.py` 原封不动。

---

### 第三层：模型定义层 — `model.py`

**解决的问题**：用什么样的网络结构，从前14天的时间模式中提取信息、预测第15天。

**设计思想**：

```
原始数据：[day1, day2, ..., day14]  归一化后的标量
    ↓
LSTM 层（2层，每层32个隐藏单元）
    ↓ 读取整个序列，记住每一步的信息
最后时间步输出：[h_14]  包含了前14天所有信息
    ↓
全连接层：32 → 1
    ↓
输出：day15 的预测值（归一化尺度）
```

**为什么选 LSTM**：便利店销售有明显的周期性（周末/工作日差异），LSTM 的门机制（遗忘门/输入门/输出门）能够选择性保留/忘记历史信息，比简单 RNN 更适合捕捉这种模式。

**核心设计哲学**：**模型结构超参数可配置**。`input_size`、`hidden_size`、`num_layers` 均为显式参数，便于调优和替换为 GRU、Transformer 等。

---

### 第四层：训练管道层 — `train.py`

**解决的问题**：如何高效、稳定地训练模型，得到一个可用的模型权重文件。

**设计思想**：

| 训练要素 | 选择 |
|---------|------|
| 损失函数 | MSE（均方误差），适合回归任务 |
| 优化器 | Adam（自适应学习率，无需手动调参） |
| 早停机制 | 记录验证集最佳 loss，仅在 loss 下降时保存模型 |
| GPU/CPU自适应 | `device = torch.device("cuda" if torch.cuda.is_available() else "cpu")` |
| 模型保存格式 | `state_dict()` 轻量字典，只保存权重不含结构 |

**核心设计哲学**：**训练与推理解耦**。训练管道专注"如何训练好模型"，推理管道专注"如何使用模型"，通过 `.pth` 权重文件作为两者的唯一接口。

---

### 第五层：推理管道层 — `inference.py`

**解决的问题**：模型训练完成后，如何在真实业务场景中使用模型。

**两条业务线**：

#### 业务线 A：预测明日营业额（`predict_next_day`）
- 读取最新14天数据 → 单步前向传播 → scaler反归一化 → 输出"元"
- 核心价值：每晚预测次日营业额，辅助备货决策

#### 业务线 B：增量微调（`incremental_train`）
- 用新收集的数据持续更新模型
- 核心挑战：防止"灾难性遗忘"——大学习率会让模型完全忘记旧规律
- 解决方案：**极小学习率（1e-4）+ 少量Epoch（5-10）+ 不覆盖原始baseline**

**核心设计哲学**：**模型的持续运营**。训练管道是一次性的（建立baseline），推理管道是持续运行（每日预测+定期微调）。

## 三、架构总览：模块依赖关系

```
data_processor.py        # 上游：原始xlsx → 干净CSV
        ↓
dataset.py               # 中游：CSV → PyTorch DataLoader
        ↓
model.py                  # 核心：LSTM 网络结构定义
        ↑
        |
train.py  ← ← ← ← ← ← ←  # 训练：优化器更新 model 的权重
        ↓
models/lstm_baseline.pth  # 产物：基准模型权重
models/scaler.pkl         # 产物：归一化器（推理必备）
        ↓
inference.py              # 下游：从权重文件推理或增量训练
        ↓
models/lstm_finetuned.pth  # 产物：微调后的增量模型
```

**依赖方向说明**：
- 数据流方向：`data_processor → dataset → model → train`
- 使用方向：`inference` 反向消费 `model` + `train` 的产物
- `dataset.py` 和 `model.py` 是纯粹的无状态模块，可独立测试
- `train.py` 和 `inference.py` 是有状态的执行管道

## 四、扩展性设计：如何演进到股票量化

| 当前模块 | 股票场景的改动点 |
|---------|---------------|
| `data_processor.py` | 列名映射从 `date/type/amount` 改为 `date/open/close/high/low/volume`；过滤逻辑从"收入"改为"股票买卖" |
| `dataset.py` | 多变量输入（OHLCV → 5个特征），seq_length 可能增大到30-60 |
| `model.py` | `input_size` 从1改为5；可替换为 GRU、Transformer；可加 attention 机制 |
| `train.py` | 损失函数可保留MSE；可加入学习率调度（ReduceLROnPlateau） |
| `inference.py` | 预测目标从"单变量下一天"改为"多变量下一天"或"未来N天" |

**核心保证**：上游数据处理和下游推理管道独立修改，不会互相影响。
